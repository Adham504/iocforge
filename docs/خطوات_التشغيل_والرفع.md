## 🟢 الخطوة 1: تحميل المشروع وفتح الطرفية (Terminal)

### ماذا نفعل؟
انقل مجلّد `iocforge` كاملًا إلى جهازك (مثلًا داخل `~/Projects/iocforge`)،
ثم افتح **الطرفية (Terminal)** بداخله:

```bash
cd path/to/iocforge
```

### لماذا؟
- **الطرفية** هي نافذة نكتب فيها الأوامر النصّية (بدل الضغط بالفأرة).
- الأمر `cd` اختصار لـ **C**hange **D**irectory أي "انتقل إلى المجلّد".
- نحن نخبر الطرفية: "اعمل من داخل مجلّد المشروع"، حتى تجد كل الملفات.

> 🍎 على ماك/لينكس: افتح تطبيق **Terminal**.
> 🪟 على ويندوز: افتح **PowerShell**.
> أو ببساطة: افتح المجلّد في **VS Code** ثم `Terminal > New Terminal`.

## 🟢 الخطوة 2: إنشاء بيئة افتراضية وتثبيت المشروع

### ماذا نفعل؟

**على ماك / لينكس:**
```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -e ".[parsers]"
```

**على ويندوز (PowerShell):**
```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -e ".[parsers]"
```

### لماذا كل سطر؟

**السطر الأول: `python -m venv .venv`**
- ينشئ **بيئة افتراضية** (Virtual Environment) في مجلّد اسمه `.venv`.
- **لماذا؟** تخيّل أن لديك عدة مشاريع، كل واحد يحتاج إصدارًا مختلفًا من مكتبة
  معيّنة. البيئة الافتراضية مثل "صندوق منفصل" لكل مشروع، فلا تتعارض المكتبات
  مع بعضها أو مع نظامك. هذه **أفضل ممارسة** احترافية في بايثون.

**السطر الثاني: `source .venv/bin/activate` (أو `Activate.ps1`)**
- **يُفعّل** البيئة الافتراضية.
- **لماذا؟** بعد التفعيل، أي مكتبة نثبّتها تذهب إلى الصندوق المنفصل، وأي أمر
  `python` يستخدم نسخة الصندوق. ستلاحظ ظهور `(.venv)` في بداية سطر الطرفية —
  هذه علامة أن البيئة مُفعّلة. ✅

**السطر الثالث: `pip install -e ".[parsers]"`**
- `pip` هو مدير حزم بايثون (يحمّل المكتبات من الإنترنت).
- `install -e .` يعني "ثبّت هذا المشروع في **وضع التعديل** (editable)".
  - **لماذا editable؟** لأنه يربط الأمر `iocforge` مباشرة بكودك. لو عدّلت أي
    ملف لاحقًا، يسري التعديل فورًا بدون إعادة تثبيت.
- `[parsers]` يثبّت **الإضافات الاختيارية** لقراءة ملفات PDF و DOCX و HTML
  (وهي: `pypdf` و `python-docx` و `beautifulsoup4`).

### كيف أتأكّد أنها نجحت؟
```bash
iocforge --version      # يجب أن يطبع: IOCForge 1.0.0
iocforge info           # يعرض الخدمات المُحمّلة وقدرات الأداة
```
- لو ظهر `IOCForge 1.0.0` فالتثبيت ناجح 100%.

## 🟢 الخطوة 3: إعداد مفاتيح الـ API (الإثراء)

### فكرة مهمة أولًا
كل المفاتيح **مجانية واختيارية**. بدونها الأداة تعمل وتستخرج كل المؤشرات،
لكنها **لن تستعلم** عن خطورتها من الإنترنت. لتشغيلها بدون مفاتيح أضف
`--no-enrich` لاحقًا.

### 3أ) أنشئ ملف `.env` من القالب
```bash
cp .env.example .env        # على ويندوز: copy .env.example .env
```
**لماذا؟**
- `.env.example` ملف قالب **فارغ** (آمن للرفع على GitHub لأنه بلا مفاتيح).
- `.env` هو الملف **الحقيقي** الذي ستضع فيه مفاتيحك السرّية.
- الأمر `cp` (أو `copy`) ينسخ القالب إلى ملف جديد لتعبئته.

### 3ب) احصل على المفاتيح المجانية
سجّل في كل خدمة وانسخ المفتاح من صفحة حسابك:

| الخدمة | ماذا تقدّم؟ | رابط الحصول على المفتاح |
|---|---|---|
| **VirusTotal** | نسبة الكشف، السمعة، التصنيفات | virustotal.com/gui/my-apikey |
| **AbuseIPDB** | أفضل سمعة مجانية لعناوين IP | abuseipdb.com/account/api |
| **AlienVault OTX** | النبضات، عائلات البرمجيات، مجموعات APT | otx.alienvault.com/api |
| **abuse.ch** | ThreatFox + MalwareBazaar | auth.abuse.ch |

### 3ج) الصق المفاتيح داخل `.env`
افتح ملف `.env` بأي محرّر نصوص والصق مفاتيحك:
```ini
VIRUSTOTAL_API_KEY=الصق_مفتاحك_هنا
ABUSEIPDB_API_KEY=الصق_مفتاحك_هنا
OTX_API_KEY=الصق_مفتاحك_هنا
ABUSECH_API_KEY=الصق_مفتاحك_هنا
ENABLE_ENRICHMENT=true
```

### 🔐 لماذا نستخدم ملف `.env` أصلًا؟
- **الأمان!** لو كتبنا المفاتيح داخل الكود مباشرةً ورفعناه على GitHub، سيراها
  أي شخص ويسرقها.
- الملف `.env` مذكور في `.gitignore`، أي أن Git **يتجاهله ولا يرفعه** أبدًا.
- بهذه الطريقة الكود عام (public) لكن أسرارك تبقى على جهازك فقط. هذه ممارسة
  أمنية معيارية في كل المشاريع الاحترافية.

## 🟢 الخطوة 4: اختبار أن كل شيء يعمل

### 4أ) شغّل مجموعة الاختبارات
```bash
pip install -e ".[dev]"        # يثبّت أدوات التطوير (pytest, ruff...)
pytest                          # يجب أن يظهر: 29 passed
```
**لماذا؟**
- `pytest` يشغّل **اختبارات تلقائية** كتبناها مسبقًا للتأكّد أن كل جزء يعمل
  كما يجب (المستخرِجات، القارئات، المحرّك، التقارير...).
- ظهور `29 passed` يعني أن المشروع **سليم 100%** ولم ينكسر شيء.
- هذه خطوة احترافية: لا ترفع كودًا قبل أن تتأكّد أن اختباراته تنجح.

### 4ب) جرّبها على ملف تجريبي (بدون إنترنت أولًا)
```bash
iocforge analyze examples/sample_report.txt --no-enrich
```
**لماذا `--no-enrich`؟** لتجربة الاستخراج فقط بسرعة، دون انتظار طلبات الإنترنت.

### 4ج) جرّبها مع الإثراء الحقيقي (تستخدم مفاتيحك)
```bash
iocforge analyze examples/sample_report.txt
```
**لماذا؟** الآن ستتصل الأداة بـ VirusTotal و AbuseIPDB... وتعطي كل مؤشر
تقييم خطورة. هذا يثبت أن مفاتيحك تعمل.

### 4د) افتح التقارير
ستجد التقارير في مجلّد `reports/`:
- `reports/iocforge_report.html` ← افتحه في المتصفّح (لوحة تفاعلية)
- `.json` / `.csv` / `.txt` ← نسخة للآلة / للإكسل / ملخّص سريع

جرّب على ملفاتك الخاصة:
```bash
iocforge analyze ملفي.pdf سجلّاتي.log -f html -f json -o output/
```
- `-f` يحدّد صيغة التقرير (يمكن تكرارها).
- `-o` يحدّد مجلّد الإخراج.

## 🟢 الخطوة 6: الرفع إلى GitHub

### 6أ) أنشئ مستودعًا (Repository) فارغًا على GitHub
- اذهب إلى **github.com/new**
- الاسم: `iocforge`
- **لا** تضف README أو License (لأنها موجودة لدينا بالفعل)
- اضغط **Create repository** وانسخ الرابط الذي يظهر.

### 6ب) جهّز Git وارفع المشروع
```bash
git init
git add .
git commit -m "Initial commit: IOCForge v1.0.0"
git branch -M main
git remote add origin https://github.com/اسم-المستخدم/iocforge.git
git push -u origin main
```

### شرح كل أمر ولماذا:

| الأمر | ماذا يفعل؟ |
|---|---|
| `git init` | يبدأ تتبّع المشروع بـ Git (ينشئ مجلّد `.git` مخفي) |
| `git add .` | يجهّز **كل** الملفات للحفظ (النقطة `.` تعني "كل شيء") |
| `git commit -m "..."` | يحفظ "لقطة" من الكود مع رسالة تصف التغيير |
| `git branch -M main` | يسمّي الفرع الرئيسي `main` (المعيار الحديث) |
| `git remote add origin URL` | يربط مشروعك المحلّي بمستودع GitHub |
| `git push -u origin main` | **يرفع** الكود فعليًا إلى GitHub |

> 💡 قد يطلب منك Git تسجيل الدخول. استخدم **Personal Access Token** بدل كلمة
> المرور (من GitHub: Settings > Developer settings > Personal access tokens).

### 6ج) تحقّق من النتيجة على GitHub
- افتح صفحة مستودعك وتأكّد أن ملف `.env` **غير موجود** في القائمة (يجب أن
  ترى `.env.example` فقط). ✅ هذه أهم نقطة أمان!
- ملف CI (`.github/workflows/ci.yml`) سيعمل تلقائيًا ويظهر علامة ✓ خضراء بعد
  نجاح الاختبارات.
